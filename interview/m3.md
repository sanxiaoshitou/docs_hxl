# 面试题解答

## 1、什么是覆盖索引

**覆盖索引（Covering Index）** 是指一个索引包含了查询语句中所有需要返回的字段，查
询时不需要回表查数据，直接从索引中就能获取全部所需数据。

### 核心特点
- **避免回表**：InnoDB 中，二级索引叶子节点只存储主键值；若索引未覆盖所有查询列，则需根据主键回表查聚簇索引。
- **减少 I/O**：只扫描索引树，大幅减少磁盘 I/O 次数。
- **提升性能**：尤其在大数据量场景下，性能提升显著。

### 使用场景
- 高频查询且字段固定的场景。
- 需要优化 `SELECT count(*)` 等统计查询。
- 通过 `EXPLAIN` 查看 `Extra` 列是否包含 `Using index` 判断是否使用了覆盖索引。

## 2、各种 MQ 怎么保证消息不会丢

### 2.1 消息丢失的三个阶段
1. **生产者丢失**：消息未成功发送到 MQ。
2. **MQ 自身丢失**：MQ 宕机导致内存中消息丢失。
3. **消费者丢失**：消费者拿到消息后未处理完就宕机。

### 2.2 RabbitMQ 保证消息不丢失

| 阶段 | 解决方案 |
|------|---------|
| **生产者** | 开启 **Confirm 模式**：`channel.confirmSelect()`，发送消息后等待 Broker 的 ACK 确认；失败则重试或记录日志。 |
| **MQ 自身** | 1. 队列持久化：`durable=true`<br>2. 消息持久化：`deliveryMode=2`（Persistent）<br>3. 开启镜像队列：多个节点同步备份 |
| **消费者** | 关闭自动 ACK，开启手动 ACK：`autoAck=false`，业务处理完成后再调用 `channel.basicAck()` |

### 2.3 Kafka 保证消息不丢失

| 阶段 | 解决方案 |
|------|---------|
| **生产者** | `acks=all`：等待所有 ISR 副本确认；开启重试机制 `retries>0` |
| **MQ 自身** | 1. 副本机制：`replication.factor >= 3`<br>2. `min.insync.replicas=2`：至少 2 个副本同步才认为写入成功<br>3. `unclean.leader.election.enable=false`：禁止非同步副本竞选 Leader |
| **消费者** | 关闭自动提交偏移量 `enable.auto.commit=false`，业务处理成功后手动 `commitSync()` |

### 2.4 RocketMQ 保证消息不丢失

| 阶段 | 解决方案 |
|------|---------|
| **生产者** | 1. 同步发送：`producer.send(msg)` 等待返回结果<br>2. 异步发送 + 回调重试<br>3. 事务消息：保证本地事务与消息发送的一致性 |
| **MQ 自身** | 1. 同步刷盘：`flushDiskType=SYNC_FLUSH`<br>2. 主从同步复制：`brokerRole=SYNC_MASTER`<br>3. 开启 Dledger 高可用集群 |
| **消费者** | 消费成功后手动返回 `CONSUME_SUCCESS`，失败则返回 `RECONSUME_LATER` 进入重试队列 |

## 3、各种 MQ 怎么做延迟队列

### 3.1 RabbitMQ 实现延迟队列

**方案一：死信队列（DLX）+ TTL**
- 创建一个普通队列，设置 `x-message-ttl`（消息过期时间）和 `x-dead-letter-exchange`（死信交换机）。
- 消息过期后自动路由到死信队列，消费者监听死信队列即可实现延迟消费。

**方案二：RabbitMQ 延迟消息插件（rabbitmq_delayed_message_exchange）**
- 安装插件后，声明 `x-delayed-message` 类型交换机。
- 发送消息时指定 `x-delay` 头部（单位毫秒），消息到达延迟后自动路由。

```java
// 使用插件示例
AMQP.BasicProperties props = new AMQP.BasicProperties.Builder()
    .headers(Collections.singletonMap("x-delay", 5000)) // 延迟 5 秒
    .build();
channel.basicPublish("delayed.exchange", "routing.key", props, msg.getBytes());
```

### 3.2 Kafka 实现延迟队列

Kafka **原生不支持延迟消息**，需自行实现：

**方案一：时间轮 + 延迟主题**
- 生产者将消息发送到 `delay-topic`，并附带目标执行时间戳。
- 消费者拉取消息后判断时间是否到达，未到达则重新投递或存入时间轮等待。
- 时间到达后将消息转发到 `target-topic`。

**方案二：基于 Kafka 的定时调度**
- 使用 Kafka Streams 或外部调度框架（如 Quartz），按时间分区存储消息，定时扫描转发。

**方案三：借助外部中间件**
- 如 Redis 的 ZSet（按时间排序），到期后推送到 Kafka。

### 3.3 RocketMQ 实现延迟队列

RocketMQ **原生支持延迟消息**，使用最为方便：
- 发送消息时设置 `delayTimeLevel`，支持 18 个固定延迟级别（1s, 5s, 10s, 30s, 1m, 2m, 3m, 4m, 5m, 6m, 7m, 8m, 9m, 10m, 20m, 30m, 1h, 2h）。
- RocketMQ 5.0 后支持任意时间精度的定时消息。

```java
Message msg = new Message("TopicTest", "TagA", "OrderID", "body".getBytes());
// 设置延迟级别，3 对应 10 秒
msg.setDelayTimeLevel(3);
producer.send(msg);
```

### 3.4 对比总结

| MQ | 延迟队列支持 | 实现方式 |
|----|-----------|---------|
| RabbitMQ | 需配置 | 死信队列 + TTL / 延迟插件 |
| Kafka | 不支持原生 | 时间轮、外部调度、Redis ZSet |
| RocketMQ | 原生支持 | 内置延迟级别 / 定时消息 |

## 4、分布式事务技术

### 4.1 分布式事务产生背景
单体系统拆分为微服务后，一个业务操作可能涉及多个服务的多个数据库，需要保证跨服务的数据一致性。

### 4.2 常见解决方案

#### ① 两阶段提交（2PC / Two-Phase Commit）
- **协调者** 询问所有 **参与者** 是否可以执行提交（Phase 1：Prepare）。
- 所有参与者返回 Yes 后，协调者发送 Commit 指令（Phase 2：Commit）。
- 若有任一参与者返回 No，则发送 Rollback。
- **缺点**：同步阻塞、单点故障、数据不一致风险（协调者宕机）。

#### ② 三阶段提交（3PC）
- 在 2PC 基础上增加 **CanCommit** 预询问阶段，引入超时机制。
- 降低了阻塞时间，但实现复杂，网络分区时仍可能不一致。

#### ③ TCC（Try-Confirm-Cancel）
- **Try**：预留资源，执行业务检查。
- **Confirm**：确认执行业务，真正提交。
- **Cancel**：取消执行，释放预留资源。
- **特点**：业务侵入性强，需为每个操作编写三阶段逻辑；性能较好，无全局锁。
- **框架**：ByteTCC、TCC-Transaction。

#### ④ Saga 模式
- 将长事务拆分为多个本地事务，每个本地事务提交后立即释放资源。
- 若某步失败，执行 **补偿操作**（Compensating Transaction）回滚前面已完成的步骤。
- **编排方式**：
  - **编排式 Saga（Choreography）**：各服务完成本地事务后发送事件，由事件驱动下一步。
  - **协调式 Saga（Orchestration）**：由 Saga 协调器统一调度各服务的执行和补偿。
- **框架**：Seata Saga、Axon Framework、Eventuate。

#### ⑤ 本地消息表（ eventual consistency ）
- 在业务数据库中增加 **消息表**，业务操作与消息写入放在同一本地事务中。
- 后台定时任务扫描消息表，将消息发送到 MQ。
- MQ 消费成功后删除或标记消息。
- **特点**：最终一致性，实现简单，适合对一致性要求不高的场景。

#### ⑥ MQ 事务消息
- 以 RocketMQ 为例：
  1. 生产者发送 **半消息**（Half Message），对消费者不可见。
  2. 生产者执行本地事务。
  3. 本地事务成功，发送 Commit，消费者可见；失败则发送 Rollback。
  4. Broker 提供 **回查机制**，检查本地事务状态。
- **特点**：实现最终一致性，RocketMQ 原生支持。

#### ⑦ Seata（阿里开源分布式事务框架）
Seata 提供 **AT、TCC、Saga、XA** 四种模式：

| 模式 | 原理 | 特点 |
|------|------|------|
| **AT 模式** | 基于 SQL 解析，自动生成反向 SQL 做回滚（Undo Log） | 无侵入，使用最广泛 |
| **TCC 模式** | 业务实现 Try/Confirm/Cancel 三阶段 | 高性能，侵入性强 |
| **Saga 模式** | 状态机引擎编排长事务，失败时执行补偿 | 适合业务流程长、需状态可视化 |
| **XA 模式** | 基于 XA 协议的 2PC | 强一致性，阻塞性能差 |

- **核心组件**：
  - **TC（Transaction Coordinator）**：事务协调器，维护全局事务状态。
  - **TM（Transaction Manager）**：事务管理器，定义全局事务范围。
  - **RM（Resource Manager）**：资源管理器，管理分支事务资源。

### 4.3 方案选型建议

| 场景 | 推荐方案 |
|------|---------|
| 强一致性，短事务，低并发 | XA / 2PC |
| 高并发，允许短暂不一致，需快速响应 | TCC、Saga |
| 无侵入，快速集成，一般业务 | Seata AT 模式 |
| 最终一致性，系统解耦 | 本地消息表、MQ 事务消息 |
| 长流程业务（如审批、物流） | Saga 模式 |

---

> 总结：覆盖索引的核心是**减少回表**；MQ 防丢失要从**生产者、Broker、消费者**三端入手；延迟队列优先选 RocketMQ 原生支持，RabbitMQ 可用插件或死信队列，Kafka 需自行实现；分布式事务没有银弹，需根据业务对一致性和性能的要求选择合适方案。
