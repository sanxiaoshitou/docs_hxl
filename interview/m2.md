## 1、慢 SQL 主要体现在哪几个方面，原来是多少秒改进后是多少秒

**体现方面：**
- 查询响应时间过长（如 > 1s）
- CPU/IO 飙升，数据库连接池耗尽
- 接口超时，前端页面卡顿

**优化前后对比（举例）：**

| 场景 | 优化前 | 优化后 |
|------|--------|--------|
| 无索引全表扫描 | 5~10s | 50~200ms |
| 大分页深度查询 | 8s | 100ms（改用游标/覆盖索引） |
| 多表 JOIN 未优化 | 3~5s | 100~300ms |

## 2、StringBuilder 与 String 的区别

| 特性 | String | StringBuilder |
|------|--------|---------------|
| 可变性 | 不可变（final char[]） | 可变（char[] 可扩容） |
| 线程安全 | 线程安全（不可变即安全） | 非线程安全 |
| 性能 | 拼接时产生大量中间对象 | 单线程拼接效率高 |
| 使用场景 | 字符串常量、少量操作 | 单线程大量字符串拼接 |

## 3、String 不可变，为什么可以重新赋值

```java
String s = "a";
s = "b";  // 这是允许的
```

- **"不可变"** 指 String 对象本身的值（底层 char[] 内容）不可变
- **"重新赋值"** 是改变引用变量 `s` 指向的地址，让其指向新的 String 对象 `"b"`
- 原对象 `"a"` 仍在常量池中（未被修改），只是 `s` 不再引用它

## 4、String +1 的赋值流程？StringBuilder 为什么不安全

**String + 的赋值流程：**

```java
String s = "a" + "b";
```

编译器优化为：

```java
String s = new StringBuilder("a").append("b").toString();
```

即：创建 StringBuilder → append → toString() 生成新 String

**StringBuilder 为什么不安全：**

- 内部 `char[] value` 没有加锁
- 多线程并发 `append()` 时，可能导致：
  - 数据覆盖（两个线程同时写入同一位置）
  - 数组扩容异常（`count` 计数不一致）
  - 最终字符串内容错乱或数组越界

---

## 5、StringBuffer 为什么安全

- 所有修改方法（`append`、`insert` 等）都用 `synchronized` 修饰
- 多线程访问时串行执行，保证数据一致性
- 代价：性能比 StringBuilder 低（锁竞争开销）

---

## 6、保证线程安全能采用哪些手段

| 手段 | 说明 |
|------|------|
| **synchronized** | JVM 内置锁，可修饰方法、代码块 |
| **ReentrantLock** | 显式锁，支持公平锁、可中断、超时获取 |
| **volatile** | 保证可见性，禁止指令重排序（不保证原子性） |
| **原子类** | `AtomicInteger`、`AtomicReference`（CAS 无锁） |
| **线程本地变量** | `ThreadLocal` 隔离线程数据 |
| **并发容器** | `ConcurrentHashMap`、`CopyOnWriteArrayList` |
| **不可变对象** | String、final 类（天然线程安全） |

## 7、Synchronized 和 Lock 的区别

| 对比项 | synchronized | Lock（ReentrantLock） |
|--------|--------------|----------------------|
| 实现层面 | JVM 层面（monitor） | API 层面（AQS） |
| 锁获取方式 | 自动获取/释放 | 手动 `lock()`/`unlock()` |
| 可中断 | 不支持 | `lockInterruptibly()` 支持 |
| 超时获取 | 不支持 | `tryLock(timeout, unit)` |
| 公平锁 | 非公平 | 可指定公平/非公平 |
| 条件变量 | 一个 `wait/notify` | 多个 `Condition` |
| 性能 | JDK6+ 优化后接近 | 竞争激烈时更灵活 |

## 8、CountDownLatch、信号量、循环栅栏在什么场景使用

| 工具类 | 核心机制 | 典型场景 |
|--------|----------|----------|
| **CountDownLatch** | 计数器减到 0 放行 | 主线程等待多个子线程完成（如启动检查） |
| **Semaphore** | 许可证池，acquire/release | 限流（如数据库连接池、接口 QPS 限制） |
| **CyclicBarrier** | 线程互相等待，达到数量一起放行 | 多线程分片计算后汇总结果（可循环使用） |

## 9、如何基于唤醒实现生产者消费者模型

```java
public class ProducerConsumer {
    private final List<Integer> list = new ArrayList<>();
    private final int capacity = 5;
    private final Object lock = new Object();

    // 生产者
    public void produce() throws InterruptedException {
        synchronized (lock) {
            while (list.size() == capacity) {
                lock.wait();  // 满了，等待消费
            }
            list.add(1);
            lock.notifyAll(); // 唤醒消费者
        }
    }

    // 消费者
    public void consume() throws InterruptedException {
        synchronized (lock) {
            while (list.isEmpty()) {
                lock.wait();  // 空了，等待生产
            }
            list.remove(0);
            lock.notifyAll(); // 唤醒生产者
        }
    }
}
```

**要点：** `while` 防止虚假唤醒，先 `wait` 后 `notifyAll`

## 10、有几种方式创建线程，Runnable 和 Callable 有什么区别

**创建线程的 4 种方式：**

1. 继承 `Thread` 类
2. 实现 `Runnable` 接口
3. 实现 `Callable` 接口（配合 FutureTask）
4. 线程池 `ExecutorService.submit()`

**Runnable vs Callable：**

| 特性 | Runnable | Callable |
|------|----------|----------|
| 返回值 | 无（void） | 有泛型返回值 |
| 异常抛出 | 不能抛出 checked 异常 | 可以抛出异常 |
| 方法 | `run()` | `call()` |
| 执行方式 | `Thread.start()` | `FutureTask` / 线程池 |

## 11、Callable 的返回值如何获取

```java
Callable<String> callable = () -> "result";
FutureTask<String> futureTask = new FutureTask<>(callable);
new Thread(futureTask).start();

// 获取返回值（阻塞直到执行完成）
String result = futureTask.get();

// 线程池方式
Future<String> future = executor.submit(callable);
String result = future.get(5, TimeUnit.SECONDS); // 带超时
```

## 12、JVM 的内存溢出如何解决

**常见 OOM 类型及解决：**

| OOM 类型 | 原因 | 解决方案 |
|----------|------|----------|
| Java heap space | 堆内存不足 | 增大 `-Xmx`；检查内存泄漏 |
| GC overhead limit | GC 回收效率低 | 优化代码，减少对象创建 |
| Metaspace | 类加载过多 | 增大 `-XX:MaxMetaspaceSize` |
| Unable to create new native thread | 线程数超限 | 限制线程池大小，检查线程泄漏 |
| Direct buffer memory | 堆外内存不足 | 限制 NIO 使用，增大 `-XX:MaxDirectMemorySize` |

**排查步骤：** `jmap -dump` 生成堆转储 → MAT/VisualVM 分析大对象

## 13、频繁 Full GC 怎么解决

**原因分析：**
- 老年代空间不足（对象过早晋升、内存泄漏）
- 大对象直接进入老年代
- System.gc() 被频繁调用

**解决方案：**

1. **增大堆内存** `-Xms` = `-Xmx` 避免动态扩缩
2. **调整晋升阈值** `-XX:MaxTenuringThreshold`
3. **优化代码** 减少大对象、长生命周期对象
4. **检查内存泄漏** 用 MAT 分析 dump 文件
5. **选择合适收集器** G1/ZGC 减少 STW
6. **监控日志** `-Xloggc:gc.log` 分析 GC 频率和耗时

## 15、Mapper interface 为什么不用实现类

**因为 MyBatis 使用动态代理生成实现类：**

```java
// 实际获取的是代理对象
UserMapper mapper = sqlSession.getMapper(UserMapper.class);
```

- MyBatis 通过 `MapperProxyFactory` 创建 JDK 动态代理
- 代理对象拦截方法调用，转给 `MapperMethod` 执行
- 根据接口全限定名 + 方法名，找到绑定的 XML SQL
- **优点：** 开发者只需定义接口，无需编写繁琐的 JDBC 代码

## 16、XML 和 Mapper 如何关联？MyBatis 的一级缓存、二级缓存有什么作用，支持什么样的场景

**XML 和 Mapper 关联方式：**

- XML 中的 `namespace` 必须等于 Mapper 接口的全限定名
- XML 中的 SQL `id` 必须等于 Mapper 接口的方法名
- 参数类型和返回类型需要匹配

**一级缓存 vs 二级缓存：**

| 特性 | 一级缓存 | 二级缓存 |
|------|----------|----------|
| 作用范围 | SqlSession（会话级） | Mapper namespace（应用级） |
| 默认开启 | 是 | 需手动配置开启 |
| 存储位置 | 内存（HashMap） | 可配置为内存/Redis/Ehcache |
| 失效时机 | SqlSession 关闭/提交/更新操作 | 任意增删改操作后清空 |
| 适用场景 | 同一会话内多次查询相同数据 | 读多写少、跨会话共享的热点数据 |

**注意事项：** 二级缓存在分布式环境下可能不一致，建议用 Redis 替代

## 17、慢 SQL 的优化，如何分析，优化技巧

**分析手段：**

- `EXPLAIN` 查看执行计划（type、key、rows、Extra）
- `SHOW PROFILES` / `performance_schema` 分析耗时
- 慢查询日志 `slow_query_log`
- `show processlist` 查看当前执行中的 SQL

**优化技巧：**

1. **加索引** 在 WHERE、JOIN、ORDER BY 字段上建索引
2. **覆盖索引** 让查询字段都在索引中，避免回表
3. **最左前缀** 联合索引按最左原则使用
4. **避免 select *** 只查需要的字段
5. **大分页优化** `limit 100000,10` 改用子查询/游标
6. **小表驱动大表** JOIN 时数据量小的放左边
7. **避免函数运算** 如 `where year(create_time)=2024`，用范围查询替代
8. **SQL 拆分** 复杂查询拆成多次简单查询
9. **读写分离** 查询走从库
10. **分库分表** 数据量过大时水平拆分

## 18、MyBatis 如何绑定

**绑定流程：**

1. **解析阶段**
   - `SqlSessionFactoryBuilder` 读取 `mybatis-config.xml`
   - `XMLConfigBuilder` 解析全局配置、Mapper 文件路径

2. **Mapper 注册**
   - `XMLMapperBuilder` 解析 `UserMapper.xml`
   - 将 `<select|insert|update|delete>` 解析为 `MappedStatement`
   - 存入 `Configuration.mappedStatements`（key = `namespace.id`）

3. **接口绑定**
   - `MapperRegistry.addMapper(Class)` 注册接口
   - 接口类名作为 namespace，与 XML 的 namespace 匹配
   - 方法名与 SQL id 匹配

4. **运行时绑定**
   - `sqlSession.getMapper(UserMapper.class)`
   - `MapperProxyFactory` 创建动态代理
   - 调用方法时，通过 `MapperMethod` 从 `Configuration` 中找到对应的 `MappedStatement` 执行 SQL
